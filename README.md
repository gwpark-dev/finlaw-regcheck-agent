# RegulationCheck — 금소법 컴플라이언스 점검 Agent

금융상품 광고·상담 문구의 **금융소비자보호법(금소법) 6대 판매규제 위반**을 근거 조항 인용과
함께 자동 점검하는 **RAG 기반 컴플라이언스 Agent**. AI는 1차 스크리닝만 하고, 최종 판단은
사람이 한다 (Human-in-the-loop).

> ### 🔗 라이브 데모 — **https://finlaw-regcheck-agent.streamlit.app/**
> 접속 비밀번호가 걸려 있습니다(시연·평가자에게 별도 전달). 미사용 시 앱이 잠들기 때문에
> 첫 접속에 콜드 스타트로 30초 정도 걸릴 수 있습니다.

## 해결하는 문제

금융사의 판매 문구 검수는 지금도 사람이 조문을 일일이 대조하는 수작업이라 느리고 누락이
생기며, 위반은 곧 과태료·영업정지 같은 제재 리스크로 이어진다. 그렇다고 범용 LLM에 "이 문구
위반이야?"라고 물으면 존재하지 않는 조항을 그럴듯하게 지어내는(환각) 답을 낸다 — 컴플라이언스
용도로는 치명적이다. RegulationCheck는 **실제로 검색된 조항 청크만 근거로 인용**하도록 강제해
1차 스크리닝을 자동화하고, 근거가 없으면 확정 판단 대신 사람 검토로 넘긴다.

## 주요 기능

- **6대 원칙별 판정 + 근거 인용** — 적합성·적정성·설명의무·불공정영업·부당권유·광고규제
  각각을 판정하고, **조문·호(號) 단위**로 위반 근거를 지식베이스 원문에서 인용한다.
- **수정안 제안** — 위반 소지가 있는 문구에 대해 개선안을 함께 제시한다.
- **개인정보 마스킹** — 어떤 API 호출보다 먼저 정규식으로 PII를 가린다(입력·로그 전 단계).
- **인용 검증(환각 차단)** — 인용된 조항이 지식베이스에 실재하는지 검증. 근거 없는 판정은
  자동으로 `NEEDS_REVIEW`로 강등한다 — 이 프로젝트가 막으려는 실패 모드다.
- **인용 중복 게이트(오탐 억제)** — 다른 원칙과 같은 행위를 근거로 삼고 독립 근거가 없는
  위반은 `NEEDS_REVIEW`로 강등해 과잉 지적을 줄인다.
- **판정 캐시(재현성)** — 같은 (입력 해시, 프롬프트 버전, 모델)이면 저장된 판정을 반환해
  도구 출력을 결정적으로 만든다. 재판정은 명시적 요청으로만, 그 사실도 로그에 남는다.
- **감사 로깅** — 전 요청을 타임스탬프·입력 해시와 함께 append-only JSONL로 기록.
- **점검 이력** — 세션 대화 맥락과 과거 점검 결과를 조회(감사 대응).
- **Streamlit UI** — 문구 입력 → 심각도별 리포트, 근거 조문 렌더링, 이력 탭.

## 성능

> ⚠️ 아래 수치는 **합성 평가셋 기준**이다(실사용 데이터 성능이 아니다 — 아래 *사용 데이터* 참조).

| 지표 | 값 | 비고 |
|---|---|---|
| 판정 정확도 | **0.876** | 인용 중복 게이트 적용 후, 평가셋 31건 / 186셀. 홀드아웃 12건에선 0.889 |
| 오탐률(정상→위반) | **0.018** | 게이트 전 0.096 → 후 0.018. **홀드아웃 0.000** |
| **위반 누락률(위반→OK)** | **0%** | 안전 지표. 강등된 건은 전부 사람 검토로 올라간다(누락 아님) |
| 검색 recall@5 | **1.00** | 평가 질의 10건(목표 ≥ 0.80) |
| 처리 시간 | **8~13초 / 건** | 원칙별 개별 LLM 호출 6회 |

정확도가 게이트 전(0.903)보다 소폭 낮은 것은, 게이트가 근거 귀속이 모호한 진짜 위반까지
안전하게 `NEEDS_REVIEW`로 강등하기 때문이다(사람 검토로 에스컬레이션). **위반을 `OK`로 놓친
경우는 0건**이라 도구로서 안전한 방향의 오차다.

## 사용 데이터

실데이터는 **일절 쓰지 않는다**. 개인정보 유출 리스크를 0으로 두고, 평가 라벨을 스스로 통제하기
위해서다. 공개 자료와 합성 데이터만 사용한다.

- **① 법령 (지식베이스)** — 국가법령정보센터 공개 원문 **5종**: 금소법 본법·시행령·감독규정,
  AI기본법, 금융 AI 가이드라인. `data/laws/`에 커밋(공개 자료, 개정 시 재수집).
- **② 점검 문구 (평가셋)** — LLM 생성 후 사람이 검수·라벨링한 **합성 데이터**.
  평가셋 31건(186셀) + 과적합 검증용 홀드아웃 12건(72셀). 실험 중 **동결**.
  라벨 기준은 `data/eval/labeling_criteria.md`.

## 개발 과정 요약

5주 빌드(W06–W10). 결정은 측정으로 내렸다.

| 주차 | 범위 |
|---|---|
| W06 | 법령 PDF 청킹·임베딩 → FAISS RAG 검색 (recall@5 1.00) |
| W07 | 6대 원칙 판정 엔진(Tool) + 점검 이력 Memory (judge v2.1 + gpt-4.1) |
| W08 | 가드레일(마스킹·인용 검증·중복 게이트) + 감사 로깅 (오탐률 0.096 → 0.018) |
| W09 | 온프레미스 아키텍처 설계 + 로컬 전환 PoC 실측 |
| W10 | Streamlit 데모 |

**핵심 실측 결정 3가지:**

1. **임베딩은 `text-embedding-3-large` (ADR-003)** — 명세가 지정한 `3-small`은 recall@5
   0.20에 그쳤다(한국어 법조문 상용구가 많아 변별 실패). `3-large`로 0.70(본법)/1.00(규제
   체인). 더 잘게 쪼개거나 어휘 하이브리드를 섞는 건 오히려 악화 — 재시도 금지.
2. **오탐의 주범은 모델이 아니라 구조 → judge v2.1 + gpt-4.1 (ADR-006)** — 기여도 분해:
   구조(2단 구성요건 판정) +0.232 / 모델 +0.080. LLM에게 verdict를 직접 묻지 않고,
   구성요건별 충족·인용까지만 받아 코드가 판정을 계산한다. 단 오탐률 목표(0.15)는 gpt-4.1이라야 통과.
3. **confidence 폐기 → 인용 중복 게이트 (ADR-007)** — 정답 위반과 오탐의 confidence가
   최소값부터 겹쳐(둘 다 0.95) 임계값으로 걸러지지 않았다. confidence 대신 **인용 중복**을
   신호로 써 오탐률 0.096 → 0.018.

주요 결정 전반은 `docs/decisions/`의 **ADR 9건**에 근거·측정치와 함께 기록돼 있다.

## 아키텍처

파이프라인(`agent/pipeline.py`의 `check()`)은 한 줄로 이렇다:

```
개인정보 마스킹 → RAG 검색(본법 조항 고정 주입 + 하위규정 유사도 보강)
   → 원칙별 판정(6회) → 인용 검증·중복 게이트 → 리포트 + 감사 로그
```

가드레일이 파이프라인을 감싼다 — 입력 마스킹은 모든 API 호출 이전, 인용 검증은 판정 이후.
`rag/`는 판정을 모르고, `agent/judge.py`는 임베딩 API를 직접 부르지 않는다(경계 분리).

```
regcheck/
├── app.py                  # Streamlit UI (W10)
├── agent/
│   ├── pipeline.py         # check() — 전체 파이프라인 오케스트레이션
│   ├── judge.py            # 6대 원칙 판정 엔진 (judge v2.1 + gpt-4.1)
│   ├── classifier.py       # 유형·상품군 분류 (기본값 제안)
│   ├── memory.py           # 점검 이력 + 판정 캐시
│   └── prompts/            # 버전 관리 프롬프트 (judge_v1.0 ~ v2.1)
├── rag/
│   ├── ingest.py           # 법령 PDF → 청킹 → 임베딩 → FAISS
│   └── retriever.py        # FAISS top-k 검색
├── guardrails/
│   ├── masking.py          # 개인정보 정규식 마스킹
│   └── verify.py           # 인용 검증 + 인용 중복 게이트
├── logging_/audit.py       # append-only JSONL 감사 로그
├── data/laws/              # 법령 원문 5종 (공개)
├── data/index/             # FAISS 인덱스 (데모용으로 커밋)
└── docs/                   # 명세서, W09 설계, W10 대본, ADR 9건
```

## 실행 방법

**직접 실행 대신 [라이브 데모](https://finlaw-regcheck-agent.streamlit.app/)를 써도 된다.**
로컬 실행은 아래와 같다.

**요구사항**: Python 3.11+, [uv](https://docs.astral.sh/uv/), OpenAI API 키.

```bash
# venv를 리눅스 파일시스템에 두어 /mnt/c(윈도우 마운트)의 느린 I/O를 피한다.
# 아래 export를 셸 프로필(~/.bashrc)에 넣어두면 매번 칠 필요가 없다. (WSL 환경 기준)
export UV_PROJECT_ENVIRONMENT=~/venvs/regcheck

uv sync                       # pyproject.toml 기준 의존성 설치
cp .env.example .env          # OPENAI_API_KEY=sk-... 채우기 (.env는 git 추적 제외)

uv run streamlit run app.py   # 데모 실행 → localhost 접속
```

지식베이스(FAISS 인덱스)는 `data/index/`에 커밋돼 있어 **바로 실행**된다. 법령 개정 등으로
재빌드가 필요하면:

```bash
uv run python -m rag.ingest              # data/laws/*.pdf → 청킹 → 임베딩 → data/index/
uv run python scripts/search_demo.py     # 검색 확인 (평가 질의 10건 + recall@5)
```

임베딩은 `data/cache/`에 캐시되어 재실행 시 API를 다시 청구하지 않는다.

## 알려진 한계

- **합성 데이터 기준 평가** — 위 성능 수치는 합성 평가셋에서 얻은 것으로, 실사용 성능을
  보장하지 않는다.
- **부당권유 원칙의 상대적 약세** — 6대 원칙 중 부당권유(제21조)가 가장 약하다(정확도
  0.74~0.75). 이미 위반인 문항에서 이웃 원칙까지 추가로 지적하는 형태이며, 순수 정상 문구
  오탐은 0건이라 검수자가 어차피 볼 건에만 발생한다(설계상 남긴 한계).
- **정규식 마스킹의 한계** — 이름·주소는 잡지 못한다(NER 미도입). 공개/합성 데이터만
  다루므로 현재 범위에선 문제되지 않으나, 실사용 전 보완 지점.
- **판정 LLM의 외부 API 의존** — 판정은 OpenAI `gpt-4.1`을 쓴다. 온프레미스 전환 시 임베딩은
  로컬 대체가 가능하나(bge-m3, recall@5 0.90) 판정 LLM은 GPU가 전제다 — 전환 경로와 실측은
  `docs/W09_온프레미스_아키텍처.md` 참조.
