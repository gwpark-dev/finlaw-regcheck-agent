## 1. 임베딩 교체 실측 (bge-m3) — ✅ 대체 가능

`scripts/w09_poc_embed.py`. 청킹(`chunk_document`)·임베딩 입력(`embed_input`)·recall
판정 로직은 production과 동일, **임베더만** text-embedding-3-large → BAAI/bge-m3.
별도 인덱스 `data/index_poc/`에 재인덱싱(기존 `data/index/`·`data/cache/` 무손상).

| 항목 | 값 |
|---|---|
| 모델 | BAAI/bge-m3 (CPU), 차원 1024 |
| 코퍼스 | 390 청크 (production과 동일) |
| **recall@5** | **0.90 (9/10)** — 기준선 3-large 1.00 |
| 게이트 | recall@5 ≥ 0.8 → **통과** |
| 모델 로드 | ~306초 (HF 다운로드 포함) |
| 코퍼스 임베딩 | 301.5초 / 390청크 (~0.77초/청크, CPU) |

**유일한 MISS**: `"이 상품은 원금이 보장되고 수익률도 확실합니다"` — 기대 조항
`금소법:제21조`(부당권유). Top-5는 제17조·감독규정 제14·17·23조·시행령 제11조로,
제21조가 조(條) 단위 Top-5에 못 듦. 이 문항이 겨냥하는 **제21조(부당권유)는 이 프로젝트의
알려진 최약 원칙**(CLAUDE.md 한계 1)이라, 임베딩 교체와 무관한 기존 난점과 겹친다.

## 2. 로컬 LLM 판정 실측 (Ollama) — ❌ CPU 서빙 불가 (OOM)

`scripts/w09_poc_judge.py`. `build_context()`(검색·임베딩)와 `judge_principle_v2()`
(LLM 호출)가 이미 분리돼 있어, judge.py를 고치지 않고 **검색은 production 3-large 고정
(F/H와 동일 조건) + 판정 LLM만 Ollama로 교체**. `phase_gate` on(= H), oracle 유형/상품군.

### 모델 · 엔드포인트

| 모델 | 크기 (`ollama list` / `du`) |
|---|---|
| qwen2.5:7b | 4.7 GB / 4.4 GiB |
| exaone3.5:7.8b | 4.8 GB / 4.5 GiB |

선택 근거: 둘 다 Ollama 레지스트리 현행본, 한국어 가능, 7.3GB RAM·CPU에서 구동 가능한
상한선(~7-8B). EXAONE=국내 온프레미스 서사 부합, Qwen=강력한 오픈 다국어 베이스라인.
Ollama의 OpenAI 호환 endpoint는 `json_schema` strict를 준수함(스모크: 스키마 유효 JSON, 22초/소형).

### qwen2.5:7b 실행 타임라인

| 시각 | 사건 |
|---|---|
| 초기(콜드) | 첫 원칙 호출이 **폭주** — 3,640+ 토큰을 6.1 t/s로 생성, 배열을 닫지 못하고 컨텍스트 4096 슬라이딩 윈도우 진입, 미종료. 중단 후 호출당 240초 상한 도입 |
| 15:15–16:25 (~70분) | 재기동 후 원칙 호출 **23건 완료**(POST 200 = 스키마 유효 JSON), 평균 **~3분/호출**(범위 2.5–4.5분) |
| ~16:00 이후 | **광고 규제(제22조, 구성요건 F1~F11 = 최대 출력)** 반복 폭주 → KV 메모리 잠식, 스왑 소진 진행 |
| 16:25:44 | 스왑 1.98GB→**0**, 가용 RAM→**78MB** (지속 하락). **워치독 OOM 감지 → 판정·모델 강제 종료** |

- **게이트 지표(정확도/오탐률/위반 누락률/스키마 준수율): 측정 불가 (OOM).** 항목 단위
  판정이 완료되기 전 종료됐고, 파이썬 stdout 블록 버퍼가 SIGKILL로 유실돼 verdict 미캡처.
- **처리 속도**: 원칙당 ~3분(150–270초). gpt-4.1(H)은 항목당 12.7초(6원칙 병렬).
  CPU에서는 병렬이 즉시 OOM이라 **직렬만 가능** → qwen 항목 ≈ 6×3분 ≈ 18분 vs H 12.7초
  = **약 85배** 지연(원칙 단위 대 H 항목으로는 ~14배).

### exaone3.5:7.8b — 미실행

qwen(4.7GB)보다 큰 모델(4.8GB)로 **동일 하드웨어에서 OOM이 실측 없이 확정**. 실행은
시스템 위험만 추가하므로, 지시(안정성 우선)에 따라 생략.

### 워치독

`/tmp` PoC 스크립트로 OOM(가용<150MB & 스왑여유<200MB 2회 연속) + stall(서버 로그
400초 무변화) 감지. 이번엔 OOM 경로가 발동, 시스템을 보호했다(가용 RAM은 이후 6.7GB 회복).

## 3. 결론 · 설계 함의

1. **임베딩과 판정의 대체 난이도는 비대칭.** 임베딩은 로컬 대체 즉시 가능(0.90),
   판정 LLM은 CPU 서빙이 "느림"이 아니라 **"불가(메모리 불안정)"**. Tier 2의 실질 관문은
   모델 선택이 아니라 **GPU 인프라**.
2. **병목은 judge_v2.1의 출력 구조.** 구성요건 전체를 finding·quote까지 채우는 비한정
   JSON 배열을 약한 로컬 모델이 닫지 못하고 폭주 → KV 메모리 잠식. Tier 2 진입 시 출력
   스키마 경량화(`checks` maxItems, quote 축약)를 GPU와 함께 검토해야 하며, 이는 판정
   품질 재검증(§7 게이트)을 다시 요구한다(스키마 변경 = 명세서 §7.3, 별도 ADR).
3. **Tier 1.5 성립.** 임베딩까지 로컬(bge-m3)로 내리면 반출면을 "판정 호출" 하나로 좁힐
   수 있다 — 대가는 recall 1.00→0.90 + 전체 재인덱싱.

**재현**: `uv run --extra poc python scripts/w09_poc_embed.py` (임베딩) /
`uv run python scripts/w09_poc_judge.py <model>` (판정, Ollama 구동 전제).
PoC 의존성은 pyproject `[poc]` 그룹으로 분리 — 본 파이프라인 무영향.
