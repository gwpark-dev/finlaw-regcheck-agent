# RegCheck On-Premise 배포 아키텍처 설계 (W09)

> 명세서 §10 W09 산출물 — 사내 LLM 대체 지점 · 네트워크 경계 · 로그 보존 정책
>
> - 버전: v1.0 (2026-07-15) / 작성: 박건우 (2025523504)
> - 전제 데이터: ADR-003/006/007의 실측 결과, W08 완료 시점 코드베이스

---

## 1. 목적과 범위

RegCheck를 금융사 내부 도구로 배포한다고 가정할 때의 아키텍처를 설계한다. 핵심 질문은 세 가지다: (1) 외부 의존 컴포넌트(판정 LLM, 임베딩 API)를 어디까지 사내로 대체할 것인가, (2) 데이터 반출을 어떤 경계에서 통제하는가, (3) 감사 로그를 어떻게 보존하는가.

본 문서는 실제 배포가 아닌 설계 문서다(5주 과제 범위, 명세서 §14). 단, 모든 판단은 W06~W08의 실측 수치를 근거로 한다.

## 2. 규제 전제

설계의 출발점이 되는 규제 환경 (2026-07 기준):

| 항목 | 내용 | 시사점 |
|---|---|---|
| 망분리 개선 로드맵 (금융위, 2024.8) | 금융회사의 생성형 AI 활용 허용, SaaS 이용 확대, 연구개발 환경 개선 | "외부 LLM API 무조건 불가" 전제가 해소됨 |
| 생성형 AI 혁신금융서비스 (2024.11~) | 규제 샌드박스로 지정 후 활용, 이후 감독규정 개정으로 제도화 추진 | 외부 API 경로는 지정/제도화 절차 + 보안 요건 충족이 조건 |
| SaaS 내부망 허용 (2026.4) | 금융보안원 평가 통과 SaaS의 내부 업무망 사용 허용 | 보안 평가라는 관문이 표준화되는 추세 |
| 인공지능기본법 (2026.1 시행) | AI 시스템의 투명성·신뢰성 의무 | RegCheck의 근거 인용 강제(NFR-02/03)·감사 로그(NFR-05)가 대응 지점 |

> ⚠️ 규제 세부 요건은 배포 시점에 금융위/금융보안원 최신 기준으로 재확인 필요. 본 문서는 설계 방향을 정하는 수준으로만 인용한다.

## 3. 배포 티어

세 가지 구성을 병렬로 제시한다. **권장은 Tier 1 (하이브리드)**.

### Tier 0 — 현행 (개발/과제 구성)

모든 컴포넌트 로컬 실행, 판정·임베딩만 OpenAI API 직접 호출. 개발·시연용이며 배포 대상 아님.

### Tier 1 — 하이브리드 (권장)

**사내에 두는 것**: 벡터 인덱스(FAISS), 임베딩 캐시, Guardrail 전체(마스킹·인용 검증·중복 게이트), 판정 캐시, 점검 이력, 감사 로그, UI.
**외부로 나가는 것**: 판정 LLM 호출(gpt-4.1)과 신규 문서 임베딩 호출만 — 보안 게이트웨이 경유.

권장 이유:

1. **품질 근거가 실측**: 오탐률 목표(≤0.15)는 gpt-4.1에서만 안정 통과(0.018, 홀드아웃 0.000 — ADR-006/007). 판정 품질이 도구 신뢰의 전부인데, 이를 포기하는 구성을 기본안으로 둘 수 없다.
2. **반출 데이터가 이미 최소화·통제됨**: 외부로 나가는 것은 [마스킹 완료된 점검 문구 + 공개 법령 조문 컨텍스트]뿐이다(W08 DoD: 마스킹 100%, 로그 잔존 0). 고객 데이터·내부 문서는 경계를 넘지 않는다.
3. **규제 경로 존재**: §2의 생성형 AI 활용 허용·제도화 경로가 외부 API 구성을 현실적 선택지로 만든다.

### Tier 2 — 완전 On-Premise (전환 경로)

판정 LLM과 임베딩을 사내 GPU 서빙 오픈웨이트 모델로 대체. 규제특례가 막히거나, 반출 통제 정책이 강경하거나, 호출량 증가로 API 비용이 사내 서빙 비용을 넘는 경우의 전환 경로.

- 판정 LLM 후보: 한국어 성능이 검증된 오픈웨이트 모델(예: EXAONE, Qwen 계열 등 — 선정 자체를 §7의 검증 절차로 결정)
- 임베딩 후보: 한국어 법률 텍스트 구별력 기준으로 선정 (ADR-003의 교훈: 모델 선택이 recall 0.20↔0.70을 갈랐다 — 반드시 실측)
- 벡터 DB: FAISS 유지 가능. 다중 사용자·권한 관리가 필요해지면 Chroma/Qdrant 등 서버형 비교 (명세서 §12)

### 티어 선택 기준

| 기준 | Tier 1 | Tier 2 |
|---|---|---|
| 판정 품질 | gpt-4.1 실측치 그대로 (오탐 0.018) | 사내 모델 검증 전까지 미지수 — §7 게이트 통과 필수 |
| 데이터 반출 | 마스킹된 문구 + 공개 조문만 반출 | 반출 0 |
| 규제 절차 | 특례/제도화 요건 충족 필요 | 절차 부담 최소 |
| 인프라 비용 | GPU 불필요, 건당 ~$0.09 | GPU 서버 + 운영 인력 (호출량 적으면 과잉) |
| 운영 복잡도 | 낮음 | 모델 서빙·업데이트 운영 부담 |

**전환 트리거**: ① 규제특례 경로 불가 판정, ② 월 판정 건수가 사내 서빙 손익분기 초과, ③ §7 검증에서 사내 모델이 품질 게이트(정확도 ≥0.75, 오탐 ≤0.15, 위반 누락 0) 통과.

## 4. 컴포넌트별 대체 지점

| 컴포넌트 | 현행 | Tier 1 | Tier 2 | 비고 |
|---|---|---|---|---|
| 판정 LLM | OpenAI gpt-4.1 | 동일 (게이트웨이 경유) | 사내 오픈웨이트 서빙 | 대체 시 judge_v2.1 프롬프트 재검증 필수 |
| 임베딩 | text-embedding-3-large | 동일 (신규 문서만, 캐시 우선) | 사내 임베딩 모델 + **전체 재인덱싱** | 임베딩 교체 = 인덱스 전면 재구축 (차원·공간 비호환) |
| 벡터 인덱스 | FAISS (로컬 파일) | 사내 서버 파일 | 동일 or 서버형 DB | 법령 원문은 공개 데이터라 민감도 낮음 |
| Guardrail/판정 캐시/Memory | 로컬 | 사내 | 사내 | 이미 외부 의존 없음 |
| 감사 로그 | 로컬 JSONL | 사내 WORM 스토리지 | 동일 | §6 |
| UI (Streamlit) | 로컬 | 내부 업무망 웹서버 | 동일 | 사내 SSO 연동은 배포 시 과제 (§14 제외 항목) |

핵심 관찰: **W08까지의 설계 덕에 외부 의존은 LLM/임베딩 API 두 지점뿐이다.** 마스킹이 API 호출 전에 실행되고(NFR-04), 인용 검증·게이트·캐시가 전부 로컬 코드라서, on-premise 전환은 "두 API의 endpoint 교체 + 품질 재검증" 문제로 축소된다.

## 5. 네트워크 경계 (Tier 1 기준)

```
[내부 업무망]                      [AI 게이트웨이 존 (DMZ)]        [외부]
┌────────────────────────────┐    ┌──────────────────────┐
│ 검수 담당자 브라우저          │    │  보안 게이트웨이        │
│   │                        │    │  - 아웃바운드 allowlist │
│   ▼                        │    │    (OpenAI endpoint만) │
│ RegCheck 앱 서버            │    │  - TLS 고정            │
│  ├ Guardrail-In (마스킹) ───┼──▶│  - 요청/응답 전문 로깅   │──▶ OpenAI API
│  ├ RAG 검색 (FAISS, 로컬)   │    │  - 마스킹 재검증        │    (판정·임베딩)
│  ├ 판정 캐시 / Memory       │    │    (2차 방어선)         │
│  ├ Guardrail-Out (검증·게이트)│   └──────────────────────┘
│  └ 감사 로그 ──▶ WORM 스토리지│
│ 벡터 인덱스 / 임베딩 캐시     │        ※ Tier 2에서는 이 존이
│ 법령 원문 (공개 데이터)       │          사내 GPU 서빙 존으로 대체
└────────────────────────────┘
```

경계 원칙:

1. **아웃바운드는 게이트웨이 단일 경로 + allowlist** — 앱 서버는 외부 직접 통신 불가. 게이트웨이만 지정된 API endpoint로 나간다.
2. **마스킹은 내부망에서 완료, 게이트웨이에서 재검증** — 개인정보 패턴이 게이트웨이에서 탐지되면 요청 차단 + 경보 (2차 방어선, defense in depth).
3. **인바운드 없음** — 외부에서 RegCheck로 들어오는 경로는 존재하지 않는다. 응답은 게이트웨이가 열어둔 세션으로만 수신.
4. **반출 데이터의 전문 로깅** — 경계를 넘는 모든 요청/응답 원문을 게이트웨이가 기록 (감사 시 "무엇이 나갔는가"를 증명하는 근거).

### 데이터 반출 목록 (Tier 1)

| 반출됨 | 반출 안 됨 |
|---|---|
| 마스킹 완료된 점검 대상 문구 | 마스킹 전 원문 (내부망 밖으로 나가지 않음) |
| 공개 법령 조문 컨텍스트 | 고객 개인정보 (마스킹 100%, W08 DoD) |
| 판정 프롬프트 (조문 구성요건 체크리스트) | 점검 이력·감사 로그 (전부 사내) |
| | 임베딩 캐시·벡터 인덱스 (사내) |

## 6. 감사 로그 보존 정책

| 항목 | 정책 | 근거 |
|---|---|---|
| 저장 방식 | append-only JSONL → 일 단위 봉인 후 WORM 스토리지 이관 | NFR-05 (변조 방지) |
| 무결성 | 일 단위 파일 해시 체인 (전일 해시를 당일 헤더에 포함) | 사후 변조 탐지 |
| 보존 기간 | **5년 권고** | 금융권 기록 보존 관행(전자금융거래 기록 보존 연한)에 준용. 확정은 배포사 준법감시 부서 협의 사항 |
| 기록 범위 | timestamp, input_hash, 마스킹 후 입력, 검색 조항 ID, 구성요건 판정, verdict, model/prompt_version/system_fingerprint, force_recheck 여부 | NFR-07 재현성 — "그때 왜 그렇게 판정했나"를 버전 단위로 복원 가능 |
| 접근 통제 | 검수 담당자: 자기 점검 이력만 / 준법감시: 전체 열람 / 삭제 권한: 없음 | 최소 권한 |
| 개인정보 | 로그에는 마스킹 후 입력만 저장 (W08 검증: 잔존 0건) | NFR-04 |

판정 캐시(ADR-007)와의 관계: 동일 입력 재요청은 캐시 반환이므로 로그에도 `cache_hit`로 기록된다. 감사 관점에서 "같은 문구는 같은 판정 + 그 사실의 기록"이 함께 보장된다.

## 7. 사내 모델 전환 검증 절차 (Tier 2 진입 게이트)

ADR-003/006에서 검증된 방법론을 그대로 재사용한다:

1. **동결 평가셋 사용**: 평가셋 31건 + 홀드아웃 12건 (라벨 동결 상태 유지)
2. **후보 모델 × judge_v2.1** 매트릭스 실측 (프롬프트 변형 금지 — 변수 통제)
3. **품질 게이트**: 정확도 ≥0.75, 오탐률 ≤0.15, **위반 누락(VIOLATION→OK) 0건**, 스키마 준수 100%
4. **임베딩 교체 시**: recall@5 ≥0.8 별도 검증 (ADR-003 절차) + 전체 재인덱싱
5. 게이트 통과 모델만 Tier 2 후보로 인정, 결정은 ADR로 기록

이 절차 자체가 본 프로젝트의 자산이다 — "모델을 바꿔도 품질을 증명할 수 있는 재현 가능한 검증 체계"가 이미 코드(scripts/judge_eval.py)와 데이터로 존재한다.

### 7.1 로컬 전환 PoC 실측 결과 (2026-07-15, ADR-009)

위 절차를 실제 로컬 구성(WSL2, CPU, RAM 7.3GB, Ollama)에 적용한 결과:

| 항목 | 결과 | 판정 |
|---|---|---|
| 임베딩 (bge-m3) | recall@5 = **0.90** (기준 3-large 1.00, 게이트 ≥0.8) | ✅ **대체 가능** — 유일한 MISS는 제21조(알려진 최약 원칙) |
| 판정 LLM (qwen2.5:7b) | 원칙당 2.5~4.5분(≈ gpt-4.1의 **85배**), 광고 규제(구성요건 F1~F11)에서 출력 배열 폭주 → KV 메모리 잠식 → **OOM 중단**. 게이트 지표 측정 불가 | ❌ **CPU 서빙 불가** |
| 판정 LLM (exaone3.5:7.8b) | 미실행 — qwen보다 큰 모델(4.8GB)로 동일 하드웨어에서 OOM이 논리적으로 확정 | ❌ (동일 사유) |

**설계 함의**:

1. **임베딩과 판정의 대체 난이도가 비대칭이다.** 임베딩은 로컬 대체가 즉시 가능하지만(0.90), 판정 LLM은 CPU 서빙이 "느림"이 아니라 "불가(메모리 불안정)"다. Tier 2의 실질 관문은 모델 선택이 아니라 **GPU 인프라**다.
2. **judge_v2.1의 출력 구조가 로컬 전환의 병목이다.** 구성요건 전체를 finding·quote까지 채우는 비한정 JSON 배열을 약한 모델이 닫지 못하고 폭주한다. Tier 2 진입 시 출력 스키마 경량화(checks maxItems, quote 축약)를 GPU 도입과 함께 검토해야 하며, 이는 판정 품질 재검증(§7 게이트)을 다시 요구한다.
3. **하이브리드 권장(§3)의 실측 근거 보강**: Tier 1에서도 임베딩은 bge-m3로 내릴 수 있어(반출 데이터에서 신규 문서 임베딩 호출 제거), "판정 LLM만 외부"로 반출면을 더 좁힌 **Tier 1.5** 변형이 성립한다. 단 recall 1.00→0.90 하락과 전체 재인덱싱 비용은 감수 항목.

## 8. 비용 개요 (Tier 1 vs Tier 2)

| 항목 | Tier 1 | Tier 2 |
|---|---|---|
| 판정 비용 | ~$0.09/건 (gpt-4.1, 6원칙 합계) — 월 1,000건 기준 ~$90 | API 비용 0, GPU 서버 상각 + 전력 + 운영 인력 |
| 초기 투자 | 게이트웨이 구축 (기존 보안 인프라 활용 가능) | GPU 서버 (모델 크기에 따라 수백만~수천만 원) + 모델 검증 비용 |
| 손익분기 감각 | — | 월 수만 건 이상이거나 반출 통제가 비용보다 우선일 때 |

내부 검수 도구의 현실적 호출량(월 수백~수천 건)에서는 Tier 1이 압도적으로 경제적이다. Tier 2는 비용이 아니라 **정책(반출 통제)이 강제할 때** 선택된다.

## 9. 리스크

| 리스크 | 대응 |
|---|---|
| 규제특례/제도화 요건 미충족 → 외부 API 불가 | Tier 2 전환 경로 + §7 검증 절차가 준비돼 있음 |
| OpenAI 모델 단종/변경 (gpt-4.1 의존) | 판정 캐시 키에 모델 버전 포함 — 변경 감지 시 §7 게이트 재실행. ADR-006의 기여도 분해가 "구조가 몸통"임을 보였으므로 모델 교체 충격은 제한적 |
| 사내 모델 품질 미달 (Tier 2) | 게이트 미통과 시 전환 보류 — 품질 없는 반출 통제는 도구 불신(명세서 §13)으로 귀결 |
| 법령 개정 시 재인덱싱 누락 | 법령 시행일 메타데이터(ADR-001) + 개정 모니터링을 운영 절차에 포함 |
| 게이트웨이 장애 | 판정 캐시가 기존 문구 재점검은 흡수, 신규 판정은 큐잉 후 재시도 |
