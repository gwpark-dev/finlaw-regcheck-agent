# ADR-005: 판정 엔진 — 본법 고정 주입 + 유사도 보강, 원칙별 개별 호출

- **Status**: Accepted
- **Date**: 2026-07-13

## 배경 (Context)

ADR-002/004의 미결 사항 정산. W06 실측에서 본법 제20·21조는 호가 6~7개씩 한
덩어리라 임베딩이 희석돼 무필터 유사도 검색 Top-5에서 하위규정에 밀렸다.
판정 엔진(W07)은 원칙별 판정 시 본법 조항을 항상 확보해야 하고(원칙의 정의
자체가 본법에 있음), 동시에 실무 판단 기준인 시행령·감독규정도 근거로
제공해야 한다(ADR-004의 규제 체인 구조).

## 대안 검토 (Alternatives Considered)

### 검색 방식

- **본법 고정 주입 + 유사도 보강 (채택)**: 원칙별 본법 조항(태그 기반,
  신뢰도 100%)은 검색 없이 컨텍스트에 상시 포함, 유사도 검색은 하위규정
  발굴 전담 — 본법 누락 0% 보장, 구현 단순. 컨텍스트 길이 증가가 비용.
- **2-pass 검색 병합**: principle_filter 검색 + 무필터 검색 병합 — 본법도
  유사도 순위에 의존해 희석 문제가 완전히 해결되지 않음.
- **조문 대응표**: 가장 정확하나 구축·유지 비용으로 W06에서 이미 기각 (ADR-002).

### LLM 호출 구조

- **원칙별 개별 호출 (채택)**: 원칙당 1회, 최대 6회 — 원칙별 집중된
  컨텍스트로 판정 정확도 유리, 원칙 단위 JSON 스키마 단순. 비용·지연 6배가
  단점이나 gpt-4o-mini 기준 건당 수 원 미만, 병렬화 시 NFR-01(30초) 충족.
- **단일 호출**: 저렴하나 긴 컨텍스트에서 원칙 간 판단 희석·누락 위험.
- **2단계(스크리닝→정밀)**: 효율적이나 1차 누락 시 복구 불가 — 컴플라이언스
  도구에서 위험한 오답 구조.

## 결정 (Decision)

**판정 엔진은 원칙별로 [본법 조항 고정 주입 + 하위규정 유사도 보강 검색]
컨텍스트를 구성하고, 원칙당 1회씩 개별 LLM 호출로 판정한다.** 임베딩 검색이
약한 부분(본법)은 결정적(deterministic) 주입으로 대체하고, 검색은 잘하는
일(하위규정 발굴)만 시킨다.

## 영향 (Consequences)

- ✅ 본법 근거 누락 구조적으로 0% — ADR-002/004 미결 사항 해소
- ✅ 원칙별 판정 프롬프트를 독립적으로 버전 관리·개선 가능 (agent/prompts/)
- ⚠️ 입력 1건당 LLM 최대 6회 호출 — 지연 누적 시 병렬 호출로 전환 (NFR-01 30초 기준)
- ⚠️ 보강 검색 결과가 판정 대상 원칙과 무관할 수 있음 — 프롬프트에 "제공된
  조항 중 관련 있는 것만 인용" 지시 필요, 무관 조항 무시 능력은 평가셋으로 검증
- ⚠️ W08 인용 검증 Guardrail은 원칙별 컨텍스트(주입+검색 합집합)를 검증 기준으로 사용

## 참고 (Notes)

ADR-002의 "W07 재검토" 및 ADR-004의 "principle_filter로 메울 것" 항목은 본
ADR로 정산됨. 6개 원칙 전부 판정할지, classifier 결과로 관련 원칙만 판정할지는
W07 구현 중 평가셋 정확도를 보고 결정 — 변경 시 본 ADR 상태 갱신이 아닌
구현 파라미터 수준으로 처리.
