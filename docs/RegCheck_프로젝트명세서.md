## 1. 프로젝트 개요

### 1.1 배경

금융상품 광고문구와 상담 스크립트는 금융소비자보호법(금소법)의 6대 판매규제를 준수해야 하며, 위반 시 최대 1억원 과태료 및 일부 규제는 징벌적 과징금(관련 수입의 최대 50%) 대상이다. 현재 검수는 수작업 중심이라 물량이 많고 판단이 주관적이어서 누락 위험이 크다. 범용 LLM은 조문을 지어내는 환각(hallucination) 문제로 규제 업무에 그대로 쓸 수 없다.

### 1.2 목적

광고/상담 문구를 입력하면 **6대 판매규제 위반 소지를 근거 조항 인용과 함께 판정하고 수정안을 제시**하는 금융사 내부용(B2B) RAG Agent를 구축한다. AI는 1차 스크리닝 역할이며 최종 판단은 사람이 한다(Human-in-the-loop).

### 1.3 범위

| 구분 | 포함 | 제외 |
|---|---|---|
| 점검 대상 | 광고 문구, 상담 스크립트(텍스트) | 이미지/영상 광고, 실시간 음성 |
| 규제 | 금소법 6대 판매규제 | 자본시장법 등 타 법령 전면 커버 |
| 사용자 | 사내 상품/마케팅 담당자, 준법감시 부서 | 일반 금융소비자(대고객) |
| 데이터 | 공개 법령 원문 + 합성 스크립트 | 실제 고객 개인정보 포함 데이터 |

## 2. 점검 대상 규제 (도메인 명세)

RAG 지식베이스와 판정 체크리스트의 기준이 되는 금소법 6대 판매규제:

| # | 원칙 | 근거 조항 | 징벌적 과징금 대상 |
|---|---|---|---|
| 1 | 적합성 원칙 | 금소법 제17조 | X |
| 2 | 적정성 원칙 | 금소법 제18조 | X |
| 3 | 설명의무 | 금소법 제19조 | O |
| 4 | 불공정영업행위 금지 | 금소법 제20조 | O |
| 5 | 부당권유행위 금지 | 금소법 제21조 | O |
| 6 | 광고 규제 | 금소법 제22조 | O |

제재 근거: 과징금 제57조, 과태료 제69조.

> ⚠️ 조항 번호는 구현 시(W06) 국가법령정보센터 원문 수집 단계에서 최신 개정본 기준으로 재확인한다.

## 3. 사용자 및 유스케이스

### 3.1 사용자

- **1차: 상품/마케팅 담당자** — 배포 전 셀프 점검으로 위반 소지 사전 제거
- **2차: 준법감시/소비자보호 부서** — 최종 검수 보조, 감사 대응 근거(점검 이력) 확보

### 3.2 핵심 유스케이스

| ID | 유스케이스 | 흐름 |
|---|---|---|
| UC-1 | 광고 문구 점검 | 문구 입력 → 유형 분류 → 관련 조항 검색 → 6대 원칙별 판정 리포트 |
| UC-2 | 수정안 제안 | 위반 소지 문구에 대해 근거 조항 기반 수정 문구 제시 |
| UC-3 | 근거 조항 질의 | "이 표현이 왜 문제인지" 조항 원문 인용으로 답변 |
| UC-4 | 점검 이력 조회 | 세션/과거 점검 결과 조회 (감사 대응) |
| UC-5 | 판정 보류 에스컬레이션 | 근거 부족/저신뢰 판정 시 사람 검토로 전달 |

## 7. RAG 파이프라인 명세

### 7.1 지식베이스 구축 (W06)

1. **수집**: 국가법령정보센터에서 금소법·시행령·감독규정 원문, 인공지능기본법(2026.1 시행) 원문, 금융위 「금융분야 AI 가이드라인」, 광고 관련 감독규정 확보 (전부 공개 데이터)
2. **청킹(chunking)**: 조(條) 단위 기본, 긴 조항은 항(項) 단위 분할. 각 청크에 메타데이터 부여 — `{법령명, 조항번호, 원칙 태그(6대 중 해당), 시행일}`
3. **임베딩**: `text-embedding-3-large` (변경 사유: 실측 recall@5 0.20 → 0.70, ADR-003 참조)
4. **적재**: FAISS 인덱스 + 메타데이터 JSON 병행 저장

### 7.2 검색·판정

- **컨텍스트 구성** (ADR-005): 원칙마다 [본법 조항 고정 주입(태그 기반, 검색 없음) + 하위규정 유사도 보강 검색(Top-k, 기본 k=5)]. 임베딩 검색이 약한 본법은 결정적 주입으로 대체하고, 검색은 하위규정 발굴만 담당한다. 보강 검색 질의에는 원칙명을 붙여 해당 원칙의 규제 체인이 올라오게 한다.
- 유사도 최고값 < 임계값(초기 0.35, 실험으로 조정) → 근거 부족으로 판정 보류
- **판정 방식**: 원칙당 1회씩 개별 LLM 호출(`gpt-4.1`). **2단 구성요건 판정**(v2.1, ADR-006) — 1단(LLM)은 조문이 열거한 구성요건별로 충족/불충족/판단불가를 문구 인용과 함께 판정하고, **2단(코드)이 1단 결과만으로 verdict를 계산한다**(LLM에게 verdict를 묻지 않는다). 조문이 결정적으로 한정한 적용 요건은 코드 게이트로 강제한다 — 제22조는 광고 국면만, 제21조제6호는 투자성 상품만.
- 생성: 검색된 조항 원문만 컨텍스트로 주입, "컨텍스트에 없는 조항 인용 금지" 시스템 프롬프트 고정

### 7.3 판정 출력 스키마 (JSON)

```json
{
  "input_hash": "sha256...",
  "input_type": "광고 | 상담",
  "product_category": "보험 | 펀드 | 대출 | ...",
  "verdicts": [
    {
      "principle": "광고 규제",
      "verdict": "VIOLATION | OK | NEEDS_REVIEW",
      "confidence": 0.87,
      "evidence": [{"law": "금융소비자보호법", "article": "제22조", "quote": "..."}],
      "reason": "손실 가능성 미고지 상태로 수익률만 강조",
      "suggestion": "수정 문구 제안 ..."
    }
  ],
  "meta": {"model": "gpt-4.1", "embedding": "text-embedding-3-large",
           "prompt_version": "v2.1", "timestamp": "...",
           "seed": 20260713, "system_fingerprint": ["fp_..."]}
}
```

`evidence`가 비어 있으면 `verdict`는 자동으로 `NEEDS_REVIEW` (NFR-02/03 강제).

## 8. 데이터 명세

| 데이터 | 출처 | 용도 | 형태 |
|---|---|---|---|
| 금소법/시행령/감독규정 원문 | 국가법령정보센터(공개) | RAG 지식베이스 | 텍스트 → 청크 |
| 금융 AI 가이드라인, 광고심의규정 | 금융위/금감원/협회(공개) | 지식베이스 보조 | 텍스트 → 청크 |
| 인공지능기본법(2026.1 시행) 원문 | 국가법령정보센터(공개) | 지식베이스 보조 — AI 시스템 투명성·신뢰성 의무 근거 (Agent 자체의 규제 준수 설계 근거로도 활용) | 텍스트 → 청크 |
| 합성 광고/상담 스크립트 | LLM 생성 + 수작업 검수 | 판정 평가셋 (위반/정상 라벨) | JSONL, 최소 60건(원칙별 10건) |
| 공개 제재/불완전판매 사례 | 금감원 보도자료(공개) | few-shot 참조 (선택) | 텍스트 |

개인정보 리스크: 전부 공개/합성 데이터 — **실데이터 미사용 원칙**을 명세 수준에서 고정.

## 12. 기술 스택 및 환경

- **언어/런타임**: Python 3.11+ (WSL2 Ubuntu / Colab 병행)
- **LLM**: OpenAI API `gpt-4.1` (판정 — 변경 사유: gpt-4o-mini는 v2.1 구조 위에서도 오탐률 0.187로 목표(0.15) 미달, gpt-4.1은 0.096, ADR-006 참조), `text-embedding-3-large` (임베딩 — 변경 사유: 실측 recall@5 0.20 → 0.70, ADR-003 참조)

- **버전 관리**: git, Conventional Commits, 작업 폴더 `~/projects/regcheck`
